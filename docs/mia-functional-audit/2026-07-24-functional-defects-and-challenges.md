## 1. 감사 결론

수정 전 화면은 시각적으로 재설계되어 있었지만 전국 지역 탐색과 정부 혜택의
핵심 계약이 완성되지 않았다. 지역 선택은 6개 고정 좌표에 제한됐고 선택
직후 장소 검색을 다시 실행하지 않아 지도·검색 결과가 잠시 다른 지역을
가리킬 수 있었다. 정부 혜택 네 프리셋은 모두 `503`을 반환해 실제 카드가
한 건도 표시되지 않았다.

로컬 교정 후에는 전국 지오코딩 결과를 지도 중심과 장소 검색 좌표에 같은
동작으로 적용한다. 혜택 인증키가 없을 때는 빈 화면 대신 정부24,
K-Startup, 소상공인24의 공식 탐색 경로를 카드로 제공한다. 빠른 연속 요청의
이전 응답이 최신 결과를 덮지 않도록 요청 소유권도 추가했다.

현재 로컬 런타임은 검증됐지만 이 문서 작성 시점에는 공개 Render URL에서
교정 버전이 검증되지 않았다.

## 3. 발견 결함과 처리 상태

### F-01. 전국 탐색처럼 보이는 고정 6개 지역

- 심각도: 높음
- 원인: 프런트엔드에 지역명과 좌표를 정적 배열로 저장했다.
- 영향: 목록 밖 지역은 검색할 수 없어 서비스 핵심 가치와 화면 문구가 달랐다.
- 교정: `/api/geocode`를 추가하고 `countrycodes=kr`로 국내 검색을 제한했다.
- 검증: 종로·춘천·여수·해운대·제주가 서로 다른 국내 좌표를 반환했다.
- 상태: 로컬 해결

### F-02. 지역 라벨과 검색 중심의 불일치

- 심각도: 치명적
- 원인: 지역 선택 이벤트가 `center` 상태만 변경하고 장소 요청을 실행하지 않았다.
- 영향: 화면의 지역명, 지도 중심, 기존 결과가 서로 다른 지역을 나타낼 수 있었다.
- 교정: 지오코딩 성공 시 좌표 저장과 `/api/places` 재요청을 하나의 흐름으로 묶었다.
- 검증: 제주 검색 직후 장소 요청에 `33.4998289, 126.5313902`가 포함됐다.
- 상태: 로컬 해결

### F-03. 정부 혜택 네 프리셋의 503 막다른 화면

- 심각도: 치명적
- 원인: 선택형 외부 API 인증키가 없으면 서버가 곧바로 `503`을 반환했다.
- 영향: 네 빠른 탐색과 직접 검색이 결과 카드 없이 종료됐다.
- 교정: 키가 없을 때 공식 기관 탐색 경로를 `200` 응답 카드로 제공한다.
- 공식 경로: 정부24 혜택알리미, K-Startup, 소상공인24.
- 상태: 로컬 해결

### F-04. 200 빈 장소 응답에서 보조 검색 미실행

- 심각도: 중간
- 원인: Nominatim이 정상 상태와 빈 배열을 반환하면 결과를 즉시 확정했다.
- 영향: Overpass에 데이터가 있어도 빈 결과로 끝날 수 있었다.
- 교정: 빠른 검색이 빈 배열이면 Overpass 보조 검색을 계속한다.
- 상태: 로컬 해결

### F-05. 빠른 연속 요청의 응답 경합

- 심각도: 높음
- 원인: 요청을 중단해도 테스트 대역이나 일부 경계 상황에서 이전 응답이 늦게
  완료되면 화면 상태를 갱신할 여지가 있었다.
- 영향: 마지막으로 누른 조건과 다른 결과가 표시될 수 있었다.
- 교정: AbortController와 현재 요청 참조가 모두 일치할 때만 상태를 갱신한다.
- 검증: 늦게 도착한 이전 혜택 응답이 최신 소상공인 결과를 덮지 않았다.
- 상태: 로컬 해결

### F-06. 자동화 셀렉터의 거짓 실패

- 심각도: 검사 신뢰도 결함
- 원인: 모임 버튼의 접근 가능한 이름에는 제목과 보조문구가 함께 포함되지만
  제목만 `exact=True`로 찾았다.
- 영향: 실제 기능을 누르지 못한 테스트가 앱 결함처럼 기록될 수 있었다.
- 교정: 실제 버튼 클래스와 포함 텍스트를 결합해 네 기능을 다시 검사했다.
- 상태: 검사기 해결

### F-07. 존재 여부 진단의 거짓 정상

- 심각도: 검사 신뢰도 결함
- 원인: 초기 Vibe Clinic 진단은 요소와 코드 구조의 존재를 중심으로 검사했다.
- 영향: 지역 좌표와 HTTP 상태가 잘못돼도 `100% OK`가 될 수 있었다.
- 교정: 브라우저 요청 파라미터, 응답 상태, 카드 개수와 실패 UI를 별도 근거로 삼았다.
- 상태: 오류 패턴 기록 완료

## 5. 구현 난제와 선택

### 5.1 지역명과 좌표의 단일 진실 공급원

지역 라벨만 바꾸거나 지도만 이동하면 다시 불일치가 생긴다. 따라서 지오코딩
결과 객체를 지도 중심, 검색 반경 중심, 현재 조건 리본에 함께 전달했다.

### 5.2 외부 API 장애와 사용자 가치의 분리

공공 API 인증 실패를 그대로 노출하면 기술적으로 정직해도 사용자는 아무것도
할 수 없다. 인증키가 없는 환경에서는 지원금을 임의로 만들어 내지 않고,
공식 포털에서 조건을 이어서 확인할 수 있는 탐색 카드만 제공한다.

### 5.3 검색 결과 0건과 기능 장애의 구분

헬스장처럼 특정 반경에 실제 데이터가 없을 수 있다. 빠른 API와 보조 API를
모두 확인한 뒤에도 없을 때만 빈 상태를 표시해 0건을 장애로 오판하지 않는다.

### 5.4 실제 결함과 테스트 결함의 구분

브라우저 자동화 실패를 바로 제품 결함으로 분류하지 않았다. 네트워크 요청이
발생했는지, DOM 셀렉터가 실제 버튼을 찾았는지 분리해 다시 재현했다.

### 5.5 비소유 변경이 많은 작업 트리

사용자 문서 이동, 설정, 패키지 파일과 이전 진단 파일이 함께 변경된 상태였다.
이번 커밋은 명시한 백엔드·프런트엔드·테스트·감사 문서만 개별 스테이징한다.

## 7. 남은 한계와 공개 배포 판정

- 공식 혜택 대체 카드는 개인 자격을 판정하는 실시간 맞춤 지원금 데이터가
  아니라 공식 포털로 연결하는 안전한 안내 데이터다.
- OpenStreetMap의 지역별 데이터 밀도에 따라 정상적인 0건이 발생할 수 있다.
- 모바일 하단 탭은 고정 내비게이션이므로 작은 화면에서 현재 콘텐츠 일부와
  겹쳐 보일 수 있다. 하단 여백으로 접근은 가능하지만 후속 사용성 검토 대상이다.
- 외부 지오코딩과 장소 서비스의 이용량 정책을 지키기 위한 캐시·호출 제한은
  운영 트래픽 증가 전에 강화해야 한다.
- Render 자동 배포 완료 후 공개 URL에서 같은 전수 검사를 다시 실행해야 한다.

