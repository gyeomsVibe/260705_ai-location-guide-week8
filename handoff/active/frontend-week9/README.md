# 9주차 프론트엔드 병렬 작업 Handoff Pack

## 목적

기존 카카오맵 위치 안내 화면에 **공공 상가정보 기반 주변 카페 UI**를 추가한다. 이 작업은 프론트엔드만 담당한다. 공공 API 인증키·외부 API 호출·Render 설정은 백엔드 담당 범위다.

## 기준 상태

- 기준 브랜치: `main`
- 기준 커밋: `89931a8`
- 프론트엔드 파일: `frontend/public/index.html`
- 백엔드 파일: `backend/server.js`
- 현재 지도: Kakao Maps JavaScript SDK
- 현재 검색: 프론트엔드가 Kakao `keywordSearch()`를 직접 호출하는 2km 장소 검색
- 현재 백엔드: 정적 파일만 제공하며 `/api/cafes`는 아직 `404`

## 작업 소유권과 경계

### 수정 허용

- `frontend/public/index.html`
- 이 handoff 폴더의 문서

### 수정 금지

- `backend/**`, `render.yaml`, 루트·백엔드 `package*.json`
- `.env`, `.gitignore`, 인증키·API URL의 비밀 파라미터
- Kakao Developers 및 Render 대시보드 설정

패키지 설치는 하지 않는다. 백엔드 API를 임의로 구현하거나 공공데이터 API를 브라우저에서 직접 호출하지 않는다.

## 백엔드 계약 (구현 예정)

프론트엔드는 아래 계약만 신뢰한다. 원본 공공데이터 필드명과 서비스키를 알 필요가 없다.

```text
GET /api/cafes?lat={number}&lng={number}&radiusKm={1|3|5}&limit={1..50}&q={optional}
```

성공 응답:

```json
{
  "items": [
    {
      "id": "store-id",
      "name": "카페 이름",
      "category": "커피·음료",
      "address": "주소",
      "latitude": 37.5665,
      "longitude": 126.9780,
      "distanceKm": 0.42
    }
  ],
  "count": 1
}
```

실패 응답:

```json
{
  "error": {
    "code": "PUBLIC_API_ERROR",
    "message": "카페 데이터를 불러오지 못했습니다."
  }
}
```

`radiusKm=all` 또는 검색어 없는 전국 조회는 프론트엔드가 요청하지 않는다. 전국 모드는 후속 API 계약이 확정될 때 별도 구현한다.

## 구현 순서

### F1. 화면 구조와 상태

1. 기존 `2. 주변 실시간 검색 (반경 2km)` 영역을 `내 주변 카페 찾기`로 바꾼다.
2. 검색창 아래에 `1km`, `3km`, `5km` 반경 칩을 추가하고 기본값을 `1km`로 둔다.
3. 결과 수, 위치 상태, 로딩·빈 결과·오류 안내 영역을 추가한다.
4. 현재 카카오 키워드 검색 기능은 API 연결이 검증될 때까지 삭제하거나 기본 경로를 대체하지 않는다.

### F2. 공공 카페 렌더링 어댑터

1. `renderCafeResults(items)`를 만든다.
2. 공공 카페 항목마다 카카오 마커와 결과 카드를 하나씩 만든다.
3. 카드 클릭과 마커 클릭은 같은 선택 상태를 공유한다.
4. 카드에는 `name`, `category`, `address`, 포맷된 거리만 표시한다.
5. 외부 데이터는 `innerHTML` 문자열에 삽입하지 않는다. `textContent`와 DOM 생성 API를 사용한다.
6. 기존 `searchMarkers`, `customOverlays`를 재사용하되 카페 전용 리소스와 기존 키워드 검색 리소스가 섞이지 않게 정리 함수를 분리한다.

### F3. 위치·반경 시각화

1. `selectedPlace`의 위도·경도를 중심으로 현재 위치 마커를 갱신한다.
2. 선택한 반경의 카카오 원(circle)을 하나만 표시한다.
3. 칩을 바꾸면 원, 결과 수, 마커, 목록을 같은 요청 세대에서 함께 갱신한다.
4. 위치 권한 거부·미지원 때 기존 기본 위치를 사용하고 쉬운 안내 문구를 표시한다.

### F4. API 연결 준비

1. `fetchCafes({ lat, lng, radiusKm, query })`를 만든다.
2. `AbortController`와 요청 세대 번호로 이전 요청의 늦은 응답을 무시한다.
3. `response.ok`가 아니면 오류 화면으로 전환한다.
4. API가 현재 `404`이므로, 백엔드 담당자가 완료 신호를 주기 전에는 자동 초기 호출을 켜지 않는다.
5. 백엔드 API가 준비되면 F1~F3 UI가 `fetchCafes` 결과를 사용하도록 한 번에 연결한다.

## UX·접근성 기준

- 칩은 실제 `button`이며 선택 상태는 `aria-pressed`로 표현한다.
- 로딩·오류·결과 수 영역은 `aria-live="polite"`를 사용한다.
- 모바일에서는 검색창과 칩이 지도보다 위에 온다.
- 긴 상호명·주소는 줄바꿈되며 화면을 넘지 않는다.
- 빈 화면 대신 항상 다음 중 하나를 표시한다: 위치 확인 중 / 데이터 로딩 중 / 결과 있음 / 결과 없음 / 위치 권한 거부 / 데이터 오류.

## 완료 검증

1. `node --check`로 인라인 스크립트 문법을 검증한다.
2. 로컬 서버에서 기존 지도·위치 선택·기존 검색이 유지되는지 확인한다.
3. 백엔드 API 완료 뒤에는 1km→3km→5km에서 원·마커·목록·결과 수가 함께 바뀌는지 확인한다.
4. 카페 이름에 `<img>` 같은 문자열이 들어와도 HTML로 실행되지 않는지 확인한다.
5. `backend/**`, `.env`, 패키지 파일, Render 설정이 수정되지 않았는지 `git diff --name-only`로 확인한다.

## 인계 조건

- 프론트엔드 담당자는 변경 파일 목록, 검증 결과, 아직 API가 없어 검증하지 못한 항목을 보고한다.
- 커밋·push·배포는 이 handoff만으로 자동 실행하지 않는다. 별도 승인 후 수행한다.
